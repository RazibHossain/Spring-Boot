“How JVM manages memory?”

JVM divides memory into Heap, Stack, and Metaspace. Objects are stored in Heap and managed by Garbage Collector. Young generation handles short-lived objects with Minor GC, 
while Old generation stores long-lived objects. GC algorithms like G1 optimize pause time by collecting regions instead of the whole heap.


What is Garbage Collection?

Garbage Collection would deal with finding and deleting the garbage from the memory.

Why should a backend developer study GC?

Because in production, many “application problems” are actually memory-management problems.Your API may work perfectly in development but fail under real traffic because of:
Memory leaks
Long GC pauses
Heap exhaustion
High object creation rate
Bad caching
Inefficient JSON handling

How GC helps you diagnose these issues.

When does gc run? Any time schedule or random?
GC in Java is not random and not on a fixed schedule. It is event-driven + JVM-managed, meaning the Java Virtual Machine decides when to run it based on memory conditions. We can 
manually request gc to run but JVM may: ignore it, delay it, partially honor it.


When an object becomes eligible for garbage collector?

Nullifying the reference variable (obj = null).
Re-assigning the reference variable (obj = new Object()).
An object created inside the method (eligible after method execution).


How many types of GC Implementations
JVM has four types of GC implementations:
Serial Garbage Collector
Parallel Garbage Collector
G1 Garbage Collector
Z Garbage Collector

Serial Garbage Collector
This is the simplest GC implementation, as it basically works with a single thread. As a result, this GC implementation freezes all application threads when it runs.
Therefore, it’s not a good idea to use it in multi-threaded applications, like server environments.
1. Small applications
Best for applications with:
Small heap size
Low memory usage
Simple workloads
Example:
Small desktop apps
Command-line tools
Utility programs

Parallel Garbage Collector 
It’s the default GC of the JVM from Java 5 until Java 8 and is sometimes called Throughput Collectors. Unlike Serial Garbage Collector, it uses multiple threads for managing heap space, 
but it also freezes other application threads while performing GC.If we use this GC, we can specify maximum garbage collection threads and pause time, throughput, and footprint (heap size).


G1 Garbage Collector
G1 (Garbage First) Garbage Collector is designed for applications running on multi-processor machines with large memory space. It’s available from the JDK7 Update 4 and in later releases.
Unlike other collectors, the G1 collector partitions the heap into a set of equal-sized heap regions, each a contiguous range of virtual memory. When performing garbage collections,
G1 shows a concurrent global marking phase (i.e. phase 1, known as Marking) to determine the liveness of objects throughout the heap.After the mark phase is complete, G1 knows which
 regions are mostly empty. It collects in these areas first, which usually yields a significant amount of free space (i.e. phase 2, known as Sweeping). That’s why this method of garbage collection is called Garbage-First.
To enable the G1 Garbage Collector, we can use the following argument:
java -XX:+UseG1GC -jar Application.java


ZGC Garbage Collector
ZGC performs all expensive work concurrently, without stopping the execution of application threads for more than 10 ms, which makes it suitable for applications that require low latency. 
It uses load barriers with colored pointers to perform concurrent operations when the threads are running, and they’re used to keep track of heap usage.
Reference coloring (colored pointers) is the core concept of ZGC. It means that ZGC uses some bits (metadata bits) of reference to mark the state of the object. It also handles heaps ranging from 8MB to 16TB in size.
 Furthermore, pause times don’t increase with the heap, live-set, or root-set size.
Similar to G1, Z Garbage Collector partitions the heap, except that heap regions can have different sizes.

Red mark = Object is garbage, will be cleaned

QNA

Q1: Your service has high latency spikes every few minutes. CPU is fine. What’s your approach?
✅ Strong Answer
First suspicion: GC pauses (Stop-The-World).
Step-by-step approach:

Check GC logs:
-Xlog:gc*
Look for:
Long pause times
Frequent Full GC
Use tools:
jstat -gc
jvisualvm
If confirmed:
Switch to G1 GC
Tune:

-XX:MaxGCPauseMillis=200
🎯 Insight
Latency spike + normal CPU = usually GC or thread blocking

❓ Q2: Why does Full GC happen frequently in a well-designed system?
✅ Strong Answer

Possible reasons:

1. Memory leak
Objects retained unintentionally
2. Incorrect heap sizing
Too small Old Gen
3. Promotion failure
Young objects quickly move to Old Gen
4. High allocation rate
Too many short-lived objects
🎯 Add depth (this is what impresses interviewer)

Even if objects are short-lived, if allocation rate is very high, GC pressure increases, causing frequent cycles.

❓ Q3: Explain how G1 GC works internally
✅ Strong Answer

G1 (Garbage First) divides heap into regions instead of fixed Young/Old.

Steps:
Heap split into ~2048 regions
Tracks live data per region
Picks regions with most garbage first
Performs incremental collection
🔥 Key Features
Predictable pause time
Mixed GC (Young + Old)
Avoids full heap scan
🎯 One-liner:

G1 prioritizes regions with most garbage to minimize pause time.

❓ Q4: What causes memory leak in Java despite GC?
✅ Strong Answer

GC only removes unreachable objects, not unused ones.

Common causes:
Static collections
Cache without eviction
Listeners not removed
ThreadLocal misuse
Example:
static List<Object> cache = new ArrayList<>();
🎯 Strong insight:

Memory leak in Java = logical leak, not physical

❓ Q5: Difference between StackOverflowError and OutOfMemoryError?
✅ Answer
Error	Cause
StackOverflowError	Deep recursion, stack full
OutOfMemoryError	Heap or Metaspace full
🎯 Add depth:

Stack memory is per-thread, while heap is shared across threads.

❓ Q6: How does object allocation work in JVM?
✅ Strong Answer
Object created in Eden space
Uses TLAB (Thread Local Allocation Buffer) → fast allocation
If survives GC → moves to Survivor
Eventually → Old Gen
🎯 Key insight:

Most objects die young → optimized in Young Gen

❓ Q7: You see “GC overhead limit exceeded”. What does it mean?
✅ Strong Answer

JVM spends >98% time doing GC but recovers <2% memory.

Causes:
Memory leak
Very small heap
High object churn
Fix:
Increase heap
Fix memory leak
Optimize object creation
❓ Q8: How do you detect memory leak in production?
✅ Strong Answer
Step 1: Take heap dump
jmap -dump:live,format=b,file=heap.hprof <pid>
Step 2: Analyze
Eclipse MAT
Look for:
Dominator tree
Retained size
🎯 Key insight:

Focus on objects that retain the most memory, not just count.

❓ Q9: Why is “stop-the-world” needed? Can we avoid it?

STW ensures memory consistency during GC.

Can we avoid?

❌ No (completely)
✅ But reduce with:

❓ Q10: When would you choose ZGC over G1?
✅ Strong Answer

Use ZGC when:

Ultra-low latency required (<10ms)
Large heap (>16GB)
Real-time systems

Use G1 when:

Balanced throughput + latency
