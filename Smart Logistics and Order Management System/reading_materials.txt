Apache Kafka is a distributed streaming platform that fundamentally changes how applications handle and process streams of data. It’s not just a messaging queue, but a robust platform for handling real-time data feeds. Kafka is designed to be highly available, scalable, and fault-tolerant, making it ideal for large-scale data processing tasks. 

To truly grasp Kafka, one must understand its core components: Brokers, Topics, Producers, and Consumers. 

Brokers: Kafka operates on a cluster of servers known as brokers. These brokers are responsible for maintaining the published data. Each broker can handle terabytes of messages without impacting the performance, ensuring scalability and reliability. 
Topics: In Kafka, a topic is a category or feed name to which records are published. Topics in Kafka are multi-subscriber; they can maintain multiple consumers, making them highly versatile. They are also partitioned, meaning a topic can be split across multiple brokers to increase throughput. 
Producers: Producers are the clients or applications that publish (write) messages to Kafka topics. A producer decides which record to assign to which partition within a topic. This could be done randomly or by a specific attribute (like a key). 
Consumers: Consumers read data from brokers. They subscribe to one or more topics and process the stream of records produced to them. Kafka consumers are designed to be elastic and fault-tolerant, allowing for scalable and reliable data processing. 



Explain CAP theorem”

CAP theorem states that in a distributed system, we can only guarantee two out of consistency, availability, and partition tolerance. Since partition tolerance is unavoidable, systems usually choose between consistency and availability depending on the use case.
. CP (Consistency + Partition Tolerance)
Prioritize correct data
May reject requests
👉 Example:
Banking system

2. AP (Availability + Partition Tolerance)
Always responds
Data may be slightly inconsistent
👉 Example:
Social media feed


“Why can’t we achieve both C and A?”
Because in the presence of a network partition, a system must either reject requests to maintain consistency or accept requests and risk inconsistency. Since partitions are unavoidable in distributed systems, we cannot guarantee both consistency and availability at the same time.

Why not both C and A?
🔹 The key is Partition (P)

In real distributed systems:
Servers are on different machines
Network can fail, delay, or split
👉 This is called a network partition

🔄 What happens during a partition?
Imagine 2 nodes:
Node A  ❌ network ❌  Node B

They cannot talk to each other.
Now the problem:
A user sends a request.
Option 1: Choose Consistency (C)
Node A says:
❌ “I’m not sure what Node B has, I won’t respond”

👉 Result:

✔ Data is always correct
❌ Some requests fail

➡️ You lose Availability

Option 2: Choose Availability (A)
Node A says:
✔ “I’ll respond anyway”

👉 Result:

✔ System always responds
❌ Data might be outdated

➡️ You lose Consistency

🔥 That’s the core reason

During a partition, you cannot both:

guarantee same data everywhere (C)
and always respond (A)
🧠 Simple Analogy

Think of two ATM machines disconnected from bank server:

If both allow withdrawal → 💥 double withdrawal (inconsistent)
If one blocks → 😐 user denied (not available)

👉 You must choose:

Correct balance (C)
OR always allow withdrawal (A)




Load balancing is a technique used to distribute incoming traffic across multiple servers to ensure scalability, high availability, and reliability of a system.

Challenges and Risks of Load Balancers
Although load balancers improve performance and availability, they also introduce some challenges that must be managed properly.

Single Point of Failure: If the load balancer itself fails, it can stop traffic from reaching servers unless backup load balancers are configured.
Performance Bottleneck: If the load balancer cannot handle very high traffic, it may slow down request processing.
Configuration Complexity: Setting up load balancing correctly for large applications can be complex.
Security Risks: Since load balancers sit between users and servers, they can become targets for cyber attacks.
Cost: Hardware load balancers and high-availability configurations can increase infrastructure costs.

What is the difference between scaling out and scaling up a load balancer?
Scaling out involves adding more instances while scaling up involves adding more resources to existing instances


Load Balancer Algorithm

Round Robin (Most common)

Requests distributed one by one:

Req1 → S1
Req2 → S2
Req3 → S3
Req4 → S1

✔ Simple
❌ Doesn’t consider server load

2. Weighted Round Robin
More powerful servers get more requests

Example:

S1 (weight 3)
S2 (weight 1)

👉 S1 gets more traffic

3. Least Connections
Send request to server with fewest active connections

✔ Good for long-running requests

4. IP Hash
Same user always goes to same server

Useful for:

session-based apps
5. Random
Random distribution (simple but less efficient)

Reverse Proxy
🔹 What it does

A server that sits in front of backend servers and forwards requests.

Client → Reverse Proxy → Backend Server

But it can do MUCH more than routing.

🔹 Goal

👉 Security + abstraction + control

🔹 Responsibilities
1. Request routing
/api → service A
/auth → service B
2. Security
hides backend servers
blocks malicious requests
3. SSL termination
handles HTTPS encryption
4. Caching
serve cached responses
5. Compression / logging
improves performance
🔥 Example tools
Nginx
Apache HTTP Server

They overlap, but load balancer focuses on distributing traffic across multiple instances for scalability, while reverse proxy provides additional features like security, routing, caching, and SSL termination. In modern systems, tools like Nginx can act as both.”
